## 一、modimport 沙盒：没有 GLOBAL

**现象**：服务器启动闪退，日志 `variable 'GLOBAL' is not declared`，定位到 `modimport` 引入的脚本某行调了 `GLOBAL.xxx`。

**根因**：`modimport()` 在沙盒环境中执行脚本，不会注入 `GLOBAL` 变量。这和 `modmain.lua` 不同——`modmain.lua` 顶部有一行 `GLOBAL.setmetatable(env, ...)` 做了环境代理，所以可以直接写 `GLOBAL`，但 `modimport` 加载的文件没有这层代理。

**正确做法**：`modimport` 引入的脚本中用 `require("xxx")` 引用原版模块（Klei 的全局 `require` 已打补丁，能找到 `scripts.zip` 中的文件），不要写 `GLOBAL.require("xxx")`。

```lua
-- ❌ modimport 引入的文件中
local containers = GLOBAL.require("containers")  -- GLOBAL is nil

-- ✅
local containers = require("containers")
```

## 二、自定义 Lua 文件命名：必须唯一

**现象**：服务器启动报 `loop or previous error loading module 'containers'`。

**根因**：我们的文件叫 `scripts/containers.lua`。`modimport` 会为每个引入的文件注册一个模块，模块名取自文件名。当我们在这个文件内部调用 `require("containers")` 想引用原版模块时，Lua 发现有一个同名的、正在加载中的模块，判定为循环依赖，直接报错。

更深层的问题：即使不加 `_ly` 只是换成别的名字比如 `containers_custom.lua`，仍然可能和其他 Mod 的同名文件冲突——`modimport` 的模块注册表是全局的。

**正确做法**：自定义 Lua 文件统一使用 `<mod特征前缀>_<功能描述>_ly.lua` 格式，三层防护：

| 层级 | 示例 | 防护范围 |
|------|------|----------|
| `_ly` 后缀 | `xxx_ly.lua` | 不与原版同名；也是作者个人标记 |
| mod 特征前缀 | `portable_wardrobe_` | 不与其他 Mod 同名 |
| 功能描述 | `container_params` | 同 Mod 内不重名 |

```lua
-- ❌
-- scripts/containers.lua       → 与原版同名，循环依赖
-- scripts/containers_ly.lua    → 可能与其他 Mod 同名

-- ✅
-- scripts/portable_wardrobe_container_params_ly.lua
```

## 三、itemtestfn：slot 参数可能为 nil

**现象**：把花衬衫（夏威夷衬衫）放入便携衣柜时崩溃，日志：
```
attempt to compare nil with number
.../containers_ly.lua:84 in (method) itemtestfn
```
stack trace 显示 `slot = nil`。

**为什么 slot 会是 nil**

两条调用路径进入了 `itemtestfn`，slot 的情况不同：

**路径 A — "快速否决"（slot = nil）**：`actions.lua:2491`

玩家右键把物品拖向容器时，框架先做一个"快速否决"检查——只传 item 不传 slot：

```lua
-- actions.lua:2491
elseif not target.components.container:CanTakeItemInSlot(act.invobject) then
    -- 进不来 → 直接说 "NOTALLOWED"，不浪费时间去遍历槽位
    return false, "NOTALLOWED"
end
```

这里的语义是："这容器能否接受这种物品（不管放哪个槽）？"。框架在这个阶段根本不关心具体槽位——如果连"可能接受"都做不到，后面就不必逐槽迭代了。所以 slot 不传（`nil`）。

**路径 B — "逐槽匹配"（slot = 具体数字）**：`container.lua:406`

```lua
-- container.lua:406（GiveItem 内部）
elseif self:CanTakeItemInSlot(item, k) then
```

`k` 是 1~numslots 的迭代变量，此时 slot 一定是数字。这是我们 itemtestfn 正常工作的场景。

**那么框架自己的 CanTakeItemInSlot 是怎么处理 nil 的？**

```lua
-- container.lua:280-295
elseif slot then              -- 框架在这里 guard 了 nil
    if slot < 1 or slot > self.numslots then
        return false
    end
    -- ...
end
return self.itemtestfn == nil or self:itemtestfn(item, slot)  -- 但这里直接透传！
```

框架自己的代码处理了 nil（跳过 slot 范围检查），但最后一行把 slot **原样透传**给自定义 `itemtestfn`——它没有在调用前把 nil 替换成默认值或跳过。这是框架的一个设计疏漏：内部 guard 了，但没有在外面再 guard 一次。

**总结**：slot = nil 发生在 `actions.lua` 的"快速否决"路径，框架自己的代码对 nil 做了防御，但透传给 `itemtestfn` 时没有保护。如果你的 `itemtestfn` 对 slot 做数值运算，必须自己先判空。

**正确做法**：`itemtestfn` 第一行先判空：

```lua
local function itemtestfn(container, item, slot)
    if not slot then
        return false  -- 快速否决路径：框架在问"任意槽能放吗"，不涉及具体槽
    end
    -- 之后再做 slot 相关的数值比较
end
```

## 四、AddRecipe2：placer 参数的隐藏语义

**现象**：制作便携衣柜后产物直接部署到了地面上，捡不起来（没有 `inventoryitem` 组件）。

**根因**：`AddRecipe2` 的 `placer` 参数不是"预览模型"那么简单。**只要填了 `placer`，框架就认为这个配方的产物是"结构物"（structure）而非"物品"，进入"先建造再摆放"的流程**——产物直接出现在地面上，不经过物品栏。

这是 DST 配方系统的底层逻辑：`placer` ≠ 可选的预览美化，而是 `DEPLOYMODE` 的开关。有 placer → 结构物模式，无 placer → 物品模式。

**正确做法**：如果你的物品是通过自己的 `deployable` 组件实现"从物品栏右键部署到地面"，不要在配方里填 `placer`。`placer` 只给那些"造出来就是建筑"的东西用（比如科学机器、箱子）。

```lua
-- ❌ 物品栏版 + placer → 框架走结构物建造流程，产物直接在地面
AddRecipe2("portable_wardrobe_ly", {...}, TECH.NONE,
    { numtogive = 1, placer = "portable_wardrobe_ly_placer" })

-- ✅ 物品栏版 + 无 placer → 产物进物品栏，由 deployable 组件负责部署
AddRecipe2("portable_wardrobe_ly", {...}, TECH.NONE,
    { numtogive = 1 })
```

## 五、物品图标（recipe + inventory）的正确设置方式

借用原版图标做占位需要同时满足三个条件。下面按踩坑顺序逐一拆解。

### 5a. 第一次尝试：recipe 直接填原版路径 → resolvefilepath 报错

**代码**：
```lua
AddRecipe2("portable_wardrobe_ly", {...}, TECH.NONE,
    { atlas = "images/inventoryimages.xml", image = "wardrobe.tex" })
```

**现象**：mod 加载阶段崩溃：
```
Could not find an asset matching images/inventoryimages.xml in any of the search paths.
```

**根因**：`AddRecipe2` 的 `atlas`/`image` 在 mod 加载时通过 `resolvefilepath` 解析。该函数遍历 `package.assetpath` 中的搜索路径，用 `kleifileexists` 检查文件是否存在（`util.lua:606-612`）。mod 自身目录在搜索路径中——**前提是文件确实放在 mod 目录下**。如果 mod 的 `images/` 里没有这个文件，就找不到。

**修正**：把 `.tex`（从游戏目录）和 `.xml`（自建最小版本，仅含需要的元素坐标）放到 `mod/images/` 下。`resolvefilepath` 会通过。

### 5b. 第二次尝试：文件在但未声明 Asset → Invalid resource handle

**现象**：`resolvefilepath` 通过了，但运行时：
```
WARNING! Invalid resource handle for atlas '../mods/portable_wardrobe/images/inventoryimages.xml'
```
物品栏图标显示为石斧（默认占位），制作时崩溃。

**根因**：`resolvefilepath` 只查文件系统存在性。但引擎的**纹理资源句柄系统**是另一层——它只认通过 `Asset()` 函数在 Prefab 的 assets 表中显式声明过的文件。没声明 = 引擎不知道这个图集存在 = 无法创建纹理句柄。

**修正**：在用到该图集的每个 Prefab 的 assets 表中加一行：
```lua
Asset("ATLAS", "images/inventoryimages.xml"),
```

### 5c. 第三次尝试：inventoryitem 调 SetAtlas() → nil crash

**代码**：
```lua
inst.components.inventoryitem:SetAtlas("images/inventoryimages.xml")
```

**现象**：制作物品时崩溃：
```
attempt to call method 'SetAtlas' (a nil value)
```

**根因**：`inventoryitem` 组件的服务端实现**没有** `SetAtlas` / `SetImage` 方法。图标信息通过 net_var 属性 `atlasname` / `imagename` 传递，赋值触发回调推送到客户端 replica（`inventoryitem.lua:1-6, 72-95`）：

```lua
-- inventoryitem.lua（这是组件内部的属性机制，不是对外暴露的方法）
local function onatlasname(self, atlasname)
    self.inst.replica.inventoryitem:SetAtlas(atlasname)
end
-- ...
atlasname = onatlasname,
imagename = onimagename,
```

对外而言，**直接给属性赋值**即可：
```lua
inst.components.inventoryitem.atlasname = "images/inventoryimages.xml"
inst.components.inventoryitem.imagename = "wardrobe.tex"
```

### 完整正确方案

**① 准备文件**：从游戏目录复制 `.tex` 原版大图，自建最小 `.xml`（仅含借用的元素坐标）：

```xml
<Atlas><Texture filename="inventoryimages.tex" /><Elements>
<Element name="wardrobe.tex" u1="0.531494140625" u2="0.562255859375" v1="0.093994140625" v2="0.124755859375" />
</Elements></Atlas>
```

**② 在 Prefab 中声明 Asset**（两个 form 都要）：

```lua
local assets = {
    -- ...
    Asset("ATLAS", "images/inventoryimages.xml"),
}
```

**③ Recipe 图标**（`modmain.lua`）：

```lua
AddRecipe2("portable_wardrobe_ly", {...}, TECH.NONE,
    { atlas = "images/inventoryimages.xml", image = "wardrobe.tex" })
```

**④ Inventoryitem 图标**（Prefab 构造阶段，属性赋值）：

```lua
inst:AddComponent("inventoryitem")
inst.components.inventoryitem.atlasname = "images/inventoryimages.xml"
inst.components.inventoryitem.imagename = "wardrobe.tex"
```

### 陷阱总结

| # | 陷阱 | 现象 | 正确做法 |
|---|------|------|----------|
| 1 | 文件不在 mod 目录 | `resolvefilepath` 找不到文件 | 把 `.tex` + `.xml` 放到 `mod/images/` 下 |
| 2 | 文件在但没有 `Asset()` 声明 | `Invalid resource handle` | 在 Prefab assets 中加 `Asset("ATLAS", ...)` |
| 3 | `inventoryitem:SetAtlas()` 方法调用 | `attempt to call method (nil)` | net_var 属性直接赋值 `atlasname = "..."` |

三个条件缺一不可：文件存在 + Asset 声明 + 正确的属性/参数路径。

## 八、双 Prefab 形态转换：slot 转移的正确顺序

**关键细节**（易出错）：

1. 先读取并保存旧容器所有 slot 的内容引用
2. **清空**旧容器的 slots（置 nil，而非 RemoveItem——RemoveItem 会触发 Drop）
3. 关闭旧容器（清理 openlist）
4. 生成新实体
5. 逐槽 `GiveItem` 放回同名槽位
6. 最后才 Remove 旧实体

如果先 Remove 旧实体再转移，物品会随旧实体一起被销毁。

## 十、GetSpecificSlotForItem 的一对多匹配陷阱

**现象**：直接塞入或 shift+左键把物品放入容器时，若物品同时匹配多列（如既有保暖又有防水的装备），第一匹配槽被占后会直接失败，不尝试后续匹配槽。

**根因**：`container.lua:298-303` 的 `GetSpecificSlotForItem` 只返回**第一个 itemtestfn 通过的槽**，不检查该槽是否已被其他物品占据。而 `GiveItem`（line 523）在 `usespecificslotsforitems = true` 时跳过"任意空槽"兜底逻辑。物品卡死在匹配但被占的槽位上。

普通箱子不受影响——它们没有 `usespecificslotsforitems`，不走此函数，直接用"第一个空槽"兜底。

**修复**：在 Prefab 构造阶段覆盖此方法，加 `slots[i] == nil` 检查：

```lua
inst.components.container.GetSpecificSlotForItem = function(self, item)
    if self.usespecificslotsforitems and not self.readonlycontainer and self.itemtestfn ~= nil then
        for i = 1, self:GetNumSlots() do
            if self:itemtestfn(item, i) and self.slots[i] == nil then
                return i                             -- 匹配 且 空闲
            end
        end
    end
end
```

## 十一、地面结构物没有 Physics 组件

**现象**：部署物品到地面时崩溃：

## 十二、imagename 不能带 `.tex` 后缀

**现象**：物品栏图标不显示，客户端报 `FROMNUM` 错误。

**根因**：`inventoryitem_replica:SetImage()`（line 132）接收参数后**自动追加 `.tex`**：
```lua
function InventoryItem:SetImage(imagename)
    self.classified.image:set(imagename ~= nil and (imagename..".tex") or 0)
end
```
所以服务端赋 `imagename = "wardrobe.tex"` → 客户端变成 `"wardrobe.tex.tex"` → 图集中找不到 → 回退到 prefab 名查找 `portable_wardrobe_ly.tex` → 游戏内置图集里也没有 → `FROMNUM` 错误。

**正确做法**：
```lua
inst.components.inventoryitem.atlasname = "images/inventoryimages.xml"
inst.components.inventoryitem.imagename = "wardrobe"   -- 不带 .tex 后缀
```

## 十三、DISMANTLE 动作的 NOTEMPTY 拦截

**现象**：已部署的衣柜有内容物时右键收回被拦截，角色说 "NOTEMPTY" 台词（如温蒂的"里面并不空虚，不像我"）。

**根因**：`actions.lua:4513` 在 DISMANTLE 动作中显式检查：
```lua
if not act.target.components.container:IsEmpty() then
    return false, "NOTEMPTY"
end
```
框架假设所有便携结构物在收回前必须清空内容物。但我们的 `ChangeToItem` 会自行转移内容物，这条检查对我们没有意义。

**修复**：在部署版 Prefab 中覆盖 `IsEmpty` 始终返回 `true`，绕过拦截：
```lua
inst.components.container.IsEmpty = function() return true end
```
注意：覆盖必须在 `WidgetSetup` 之后（否则 WidgetSetup 会重置回原版方法）。

## 十四、ChangeToItem 接收 doer 参数实现收回入包

**现象**：右键收回后衣柜掉在地上，不进背包。

**根因**：`ChangeToItem` 总是用 `Transform:SetPosition` 丢在地上。但 `portablestructure:Dismantle` 会传 `doer` 进来，可以利用它让物品直接进入背包。

**修复**：函数签名改为 `ChangeToItem(inst, doer)`：
```lua
if doer ~= nil and doer.components.inventory ~= nil then
    doer.components.inventory:GiveItem(item)
else
    item.Transform:SetPosition(inst.Transform:GetWorldPosition())
end
```
锤子敲（`workable.OnFinish`）不传 doer → `nil` → 走掉落分支。

## 十五、UI 动画的布局方向陷阱

**现象**：容器 UI 显示为 3行×2列（横向2格、纵向3格），与预期的 2行×3列 不符；格子与背景错位。

**根因**：`ui_icepack_2x3` 是 2列×3行（香料包），而我们需要 3列×2行。借用 `ui_chest_3x3`（3列×3行）只取上 2 行。slotpos 坐标必须匹配动画的格子框架，使用宝箱的坐标公式（`containers.lua:1227`）：
```lua
-- 遍历 y=2,1（只取上 2 行），x=0,1,2（3 列）
for y = 2, 1, -1 do
    for x = 0, 2 do
        table.insert(slotpos, Vector3(80*x - 80, 80*y - 80, 0))
    end
end
```
y 值越大越靠屏幕上方，所以 `y=2`（head 行）在上，`y=1`（body 行）在下。

## 十六、portablestorage 标签的动画副作用：熊罐 vs 月轮宝盘

**背景**：便携衣柜最初参照熊罐（`beargerfur_sack`）模式，加了 `portablestorage` 标签 + `droponopen=true`。
结果遇到一系列问题：移动关容器、动作锁定、关闭动画异常。经过多次尝试（`skipautoclose`、覆盖 `OnUpdate`、sg 操作 `keep_pocket_rummage_mem_onexit`、手动 `GoToState("stop_pocket_rummage")`），最终发现最简方案是参照棱镜月轮宝盘：**不加标签**。

### 根因：stategraph 的 ActionHandler 路由

`SGwilson.lua:1139-1142` 是总开关：
```lua
ActionHandler(ACTIONS.RUMMAGE, function(inst, action)
    if action.invobject then
        if action.invobject:HasTag("portablestorage") and action.invobject.components.container then
            -- 路由到 "start_pocket_rummage"（持久 build 动画）
        end
    end
end)
```

带 `portablestorage` 标签 → sg 进入 `start_pocket_rummage` 状态，播放 `build_pre→build_loop`（反复弯腰翻找动画），**锁定角色动作**。同时 `container:OnUpdate` 会检测 sg 的 `keep_pocket_rummage` 标签是否消失 → 移动/被打断时自动 Close。

不带此标签 → sg 回退到默认 `doshortaction`（标准弯腰取放动画），不锁定角色动作，打开/关闭各自一次短动画。

### 两种实现的全面对比

| 维度 | 原版熊罐 | 棱镜月轮宝盘 | 我们（最终方案） |
|------|----------|-------------|-----------------|
| `portablestorage` 标签 | ✅ | ❌ | ❌ |
| `droponopen` | ✅ | ❌ | ❌ |
| OnUpdate 行为 | tag 驱动：移动/打断→关 | 默认距离检测 | 默认距离检测 |
| 开启动画 | `build_pre→build_loop` 持久弯腰 | `doshortaction` 简短弯腰 | `doshortaction` 简短弯腰 |
| 关闭动画 | 被 OnUpdate 打断时无完整动画 | `doshortaction` 简短弯腰 | `doshortaction` 简短弯腰 |
| 锁定动作 | 是（`busy` 标签） | 否 | 否 |
| 是否落地 | `droponopen` 防落地 | 不落地 | 不落地（自然行为） |
| 额外代码量 | 零（全由框架处理） | 零（全由框架处理） | 零（全由框架处理） |

### 为什么当初走了弯路

我们最初参照的模板是熊罐，但熊罐的 `portablestorage` 模式适用于**需要角色持续保持动作**的场景（如冰箱背包打开后角色蹲下翻找）。便携衣柜的使用场景则类似普通箱子——打开看一眼、关了继续走，不需要角色锁定动作。

棱镜抓住了这个本质区别，直接不设标签，利用框架默认的 `doshortaction`。

**教训**：选择参照模板时要关注**交互模式**是否匹配，而不仅是"功能类似"。熊罐和衣柜都是物品栏容器，但交互模式不同（蹲守 vs 快开）。
