# 美术相关

## PNG→TEX 转换：预乘 Alpha（Premultiplied Alpha）

### 现象

物品栏图标在游戏中周围出现白边，而不是透明。

### 根因

DST 的 KTEX 纹理格式期望 **预乘 Alpha**（RGB 值已乘以 Alpha 通道）。普通 PNG 中透明像素的 RGB 可能保持原值（如白色 255,255,255），GPU 做双线性采样时在边缘取"有色不透明像素"和"白色全透明像素"的加权平均→半白边。

ktools 的 `ktech` 默认做预乘（`--no-premultiply` 才会跳过）。

### 修复

用 Pillow 的 `ImageChops.multiply(R, A)` 逐通道预乘（内部计算 `R * A / 255`），然后再转 TEX。

**注意**：预乘必须在 Y 轴翻转（`Image.FLIP_TOP_BOTTOM`）之前做。

## scml.exe 编译缓存：不清缓存=改了 SCML 也不生效

### 现象

修改 SCML 重新编译后游戏里动画没有变化。比如两张 PNG 分别用于 `closed` 和 `open` 动画，关门的 `closed` 却一直显示开门的纹理，怎么改 SCML 都没用。

### 根因

`scml.exe` 在 `<scml目录>/anim/` 下已有旧 zip 时，会输出 "xxx.zip is up to date" 并**直接跳过重编**。引擎加载的始终是旧 build。

### 教训

**每次编译前必须手动删除旧产物**：

```bash
rm -rf exported/xxx/anim/
rm -f exported/xxx/xxx.zip
```

之后再运行 scml.exe。永远不要信任 "is up to date" 提示。

## Spriter SCML 动画：pivot 控制 sprite 锚点

### 概念

`pivot_x` 和 `pivot_y`（归一化坐标 0~1）定义 sprite 在实体世界坐标的**附着点**。实体放在地面 (x,0,z) 时：

- `pivot_y="1.0"` = sprite 最底部是附着点 → sprite 从地面向上延伸（地面结构物标准行为）
- `pivot_y="0.5"` = sprite 中心是附着点 → sprite 一半在地下
- `pivot_x="0.5"` = sprite 水平居中于附着点

### 定位过程

地面结构物需要先确认 pivot 指向艺术内容底部的正确位置，然后通过 `x`/`y` 偏移微调。注意：

- **SCML 的 `x`/`y` 是 sprite 空间坐标**，`scale_x`/`scale_y` 放大后才是游戏世界坐标。例：sprite 空间偏移 100px × scale 0.5 = 游戏世界偏移 50 单位
- `y` 正值 = sprite 向上偏移（Spriter 坐标系，不同于游戏世界坐标）

### 调试流程建议

1. 先用单图确定 pivot 位置（保持 Y 偏移为 0）
2. 确定 pivot 正确后，微调 `y` 偏移量对齐其他物体
3. 每轮只改一个值，记录变化量和效果

## autocompiler.exe 的正确使用方式

### 问题

直接在命令行运行 `autocompiler.exe <mod目录>` 往往无声失败（无输出、无生成物）。

### 正确方式

1. 将 Mod Tools 整个 `mod_tools` 文件夹复制到一个**无中文、无特殊字符**的路径（如 `C:\tmp\mod_tools`）
2. 将需要编译的 `exported/` 目录 + `modmain.lua` 放在同一个 mod 目录下
3. 从 `mod_tools` 目录内运行 `scml.exe <scml路径> <scml所在目录>`
4. 编译产物在 `<scml所在目录>\anim\xxx.zip`
5. 将该 zip 复制到 mod 的 `anim/` 目录

**之所以需要纯 ASCII 路径**：scml.exe 内部调用 Python 27 的 zipanim.py，路径含非 ASCII 字符时 subprocess 调用失败。

### 编译输出格式

合法的 `.zip` 内含：`anim.bin` + `build.bin` + `atlas-0.tex` + `atlas-1.tex` + ...。引擎通过 `.bin` 查找动画帧和 atlas 纹理。

## 工作区与游戏目录的双副本

### 教训

Mod 开发时有**两个目录**：工作区（项目管理和版本控制用）和游戏加载目录（`Steam\...\Don't Starve Together\mods\xxx\`）。修改只在工作区生效→游戏没变化→浪费大量排查时间。

### 规则

- 所有修改在工作区进行
- 修改完成后，由用户决定何时将工作区内容同步到游戏目录
- 排查"为什么改了没效果"时，第一步永远是检查游戏目录文件是否确实是新版

## SCML 文件结构最小模板

```xml
<?xml version="1.0" encoding="UTF-8"?>
<spriter_data scml_version="1.0" generator="BrashMonkey Spriter" generator_version="b5">
    <folder id="0" name="my_sprite">
        <file id="0" name="my_sprite/my_image.png" width="W" height="H" pivot_x="0.5" pivot_y="1.0"/>
    </folder>
    <entity id="0" name="my_bank">
        <animation id="0" name="idle" length="30">
            <mainline>
                <key id="0" time="0">
                    <object_ref id="0" name="s" timeline="0" key="0" z_index="0"/>
                </key>
            </mainline>
            <timeline id="0" name="s">
                <key id="0" time="0" spin="1">
                    <object folder="0" file="0" x="0" y="0" angle="0" scale_x="1" scale_y="1"/>
                </key>
            </timeline>
        </animation>
    </entity>
</spriter_data>
```

关键对应关系：

| SCML | Lua 代码 |
|------|----------|
| `entity name="xxx"` | `SetBank("xxx")` |
| `<spriter_data>` 所在文件的主名 | `SetBuild("xxx")` |
| `animation name="yyy"` | `PlayAnimation("yyy")` |
| `folder` → `file` → 路径 | 引用的 PNG 材质 |
