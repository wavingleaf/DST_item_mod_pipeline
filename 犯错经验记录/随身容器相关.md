# 随身容器相关

## 两类随身容器方案

DST 中"在物品栏中右键打开的容器"有两条实现路径，外观相似但交互模式截然不同：一条是原版熊罐/钓具箱方案（加标签，框架掌管动画和关闭），另一条是棱镜月轮宝盘方案（不加标签，走默认容器逻辑）。

| | 原版方案（熊罐/钓具箱） | 棱镜方案（月轮宝盘） |
|------|----------|-------------|
| 核心标签 | `portablestorage` | 无 |
| 开启动画 | `start_pocket_rummage`（持久弯腰翻找） | `doshortaction`（简短弯腰取放） |
| 关闭动画 | 被 `OnUpdate` 打断时无完整动画 | `doshortaction`（完整弯腰放物） |
| 动作锁定 | 是（`busy` 标签） | 否 |
| 移动时 | 自动关闭容器 | 容器保持打开 |
| 关闭条件 | 手动 X / 移动自动关 / 被打断自动关 | 手动 X / 走远距离关 |
| 代码复杂度 | 需 `droponopen=true` | 零额外参数 |
| 适用场景 | 角色蹲下持续翻找（冰箱背包、钓具箱） | 快开快关、打开后自由移动（装备收纳） |

以下详述原版方案中 `portablestorage` 标签在三个层级的硬编码行为。棱镜方案因不加标签，以下三层均不命中。

---

## `portablestorage` 标签是"合约"——加之前搜索全部引用

`portablestorage` 不只是标记"这是随身容器"，它在三个层级有硬编码行为，加标签等于同时签下三份合约：

### 1. stategraph 层：动画路由（SGwilson.lua:1138-1151）

```lua
ActionHandler(ACTIONS.RUMMAGE, function(inst, action)
    if action.invobject then
        if action.invobject:HasTag("portablestorage") then
            return "start_pocket_rummage"  -- 持久 build_pre→build_loop 弯腰翻找动画
        end
    end
    return "doshortaction"                 -- 标准弯腰取放动画
end),
```

带标签 → stategraph 进入 `start_pocket_rummage` 状态，播放持久动画，**锁定角色动作**（`busy` 标签）。
不带标签 → 回退到默认 `doshortaction`，短动画，不锁定。

### 2. component 层：自动关闭（container.lua:1075-1077）

```lua
if self.inst:HasTag("portablestorage") and
   not (opener.sg and opener.sg:HasStateTag("keep_pocket_rummage")) then
    self:Close(opener)
end
```

带标签 → OnUpdate 每帧检查，角色移动/被打断导致 sg 离开 `keep_pocket_rummage` 状态 → 自动关闭容器。
不带标签 → 此分支永远不命中，关闭只靠走远距离检测或手动 X。

### 3. action 层：落地保护（actions.lua:1067-1069）

```lua
if owner and
    (targ.components.container.droponopen) and
    not (owner:HasTag("player") and targ:HasTag("portablestorage"))
then
    -- 落地逻辑...
end
```

带标签 + `droponopen=true` → 打开时不落地，物品留在物品栏中。

### 适用场景判断

| 场景 | 加标签？ |
|------|----------|
| 角色需要蹲下持续翻找（冰箱背包、钓具箱） | ✅ |
| 快开快关、打开后角色自由移动 | ❌ |
| 打开后需要边走边看 | ❌ 绝对不能加 |

**教训**：给实体加原版标签前，必须搜索该标签在 stategraph、component、action 三层中的全部引用，理解其完整语义。标签的名称不等于它的全部行为。

## 可部署容器的双 Prefab 形态转换

> 以下问题只出现在**可部署容器**（物品版 ↔ 地面版双形态）中。纯随身容器（如熊罐、月轮宝盘）不涉及。

### slot 转移的正确顺序

从物品版部署到地面版（或反向收回）时，容器内容的转移必须严格按以下顺序：

1. 读取旧容器所有 slot 的内容引用并保存
2. **清空**旧容器的 slots（直接置 `nil`，不要调 `RemoveItem`——它会触发 Drop）
3. 关闭旧容器（清理 openlist）
4. 生成新实体
5. 逐槽 `GiveItem` 放回同名槽位
6. 最后才 `Remove` 旧实体

如果先 Remove 旧实体再转移，物品会随旧实体一起被销毁。

### `ChangeToItem` 接收 doer 参数实现收回入包

地面版右键收回时，`portablestructure:Dismantle` 会传 `doer` 进来。利用它可以判断是"收回入背包"还是"锤子敲→掉落"：

```lua
local function ChangeToItem(inst, doer)
    -- ...保存并转移内容物到新物品...
    
    if doer ~= nil and doer.components.inventory ~= nil then
        doer.components.inventory:GiveItem(item)       -- 右键收回：入包
    else
        item.Transform:SetPosition(inst.Transform:GetWorldPosition())  -- 锤子敲：掉落
    end
    
    inst:Remove()
end
```

`workable.OnFinish`（锤子敲）不传 doer → `nil` → 走掉落分支。

## DISMANTLE 动作的 NOTEMPTY 拦截

### 现象

已部署的衣柜有内容物时右键收回被拦截，角色说 "NOTEMPTY" 台词。

### 根因

`actions.lua:4513` 在 DISMANTLE 动作中硬编码检查 `container:IsEmpty()`。框架假设所有便携结构物收回前必须清空内容。但我们的 `ChangeToItem` 会自行转移内容物，这条检查没有意义。

### 修复

在地面 Prefab 中覆盖 `IsEmpty` 始终返回 `true`（必须在 `WidgetSetup` 之后）。
