# 试错经验录

> 从"开摆永鲜补丁"Mod 开发失败过程中提炼的通用经验。适用于后续 DST Mod 补丁开发参考。

| # | 类别 | 核心教训 |
|---|------|---------|
| 一 | **宏观：补丁 vs 覆盖** | `local` 函数链无法 Hook → 补丁方式有结构性天花板 |
| 二 | **加载顺序** | 本地 Mod 不按 `priority` 排序，文件系统目录顺序决定 |
| 三 | **local 闭包** | 补丁无法访问目标 Mod 的 `local` 作用域 |
| 四 | **PostInitAny 时序** | 回调按注册顺序执行，非按 Mod 加载顺序 |
| 五 | **DoTaskInTime 竞争窗口** | 延迟方案引入新的时序问题，每个"修复"需要另一个"修复"填补 |
| 六 | **TheWorld 为空** | 顶层代码不能访问 `TheWorld`，必须在回调中使用 |
| 七 | **API 溯源** | 同名概念属于不同组件，修改前必须读该组件完整源码 |
| 八 | **坏加载保护** | `Detected bad load, disabling all mods` 是 DST 保护机制，不是 Mod bug |
| 九 | **读档实体覆盖** | `AddPrefabPostInitAny` 只对新生实体有效，已存在实体用 `AddPrefabPostInit("world")` + `TheSim:FindEntities` 扫描 |
| 十 | **级联报错** | 只看第一个 LUA ERROR，忽略 `SetGlobalErrorWidget` 等连锁报错 |
| 十一 | **debug mod 设计** | 逻辑全部放回调内、统一日志前缀、独立于补丁 |

## 补充说明

**一、补丁 vs 覆盖**：直接改原 Mod 的 `changeDecorationItem`（`local` 函数），3 行代码即可保鲜。转为补丁后历经 4 版、100+ 行仍未覆盖所有时序。判别标准：列出目标 Mod 中要改的函数，若核心在 `local` 闭包链中 → 覆盖式。

**二、加载顺序**：创意工坊 Mod 按 `priority` 升序加载，本地 Mod **不按 `priority`**，按文件系统目录枚举顺序。本地测试时确保补丁文件夹名排在原 Mod 之后（如 `lay` → `z_lay_patch`）。

**三、local 闭包**：Lua 的 `local` 在编译时限定作用域，模块 A 的 `local` 对模块 B 完全不可见。补丁只能 Hook 全局锚点（`ACTIONS.X.fn`、`GLOBAL.fn`、`inst.field`），无法触及 `local` 闭包链。

**四、PostInitAny 时序**：回调按 `AddPrefabPostInitAny(cb)` 的调用顺序执行，不是按 priority。补丁先注册 → 补丁回调先跑 → 原 Mod 还没设置 `DecorationDeploy` → nil。

**五、DoTaskInTime 竞争窗口**：延迟到帧末包装，但玩家在这帧内放置了食物 → DecorationDeploy 还是裸的 → 不保鲜。"只有最后一个永鲜"即由此导致。为填补窗口又引入 ACTIONS Hook → 代码重复膨胀。

**六、TheWorld 为空**：`modmain.lua` 顶层执行时世界尚未创建，`TheWorld` 只能在回调函数内部使用。

**七、API 溯源**：`GetPerishRateMultiplier`（preserver 组件，冰箱）和 `GetLocalMultiplier`（perishable 组件，物品自身）名称相似但归属不同。修改前必读 `scripts/components/` 下完整源码。

**八、坏加载保护**：崩溃重启后 DST 自动禁用所有 Mod（`ModIndex: Detected bad load`），手动重新启用即可。
