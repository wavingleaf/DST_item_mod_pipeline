# mod_开摆_增加永鲜

## 概述

"开摆！"（Place Everything）是一个装饰品放置 Mod：玩家佩戴**不安全帽**后，可以把任意背包物品右键放置到地面上作为装饰，防止误拾取。只有戴帽子时才能取回。

**原始行为**：放置在地面上的有新鲜度物品（如食物），`perishable` 组件继续运行，新鲜度照常流失。取回时，原可堆叠物品通过 `SpawnPrefab` 重建并同步 `perishremainingtime` 来防止新鲜度被重置，但不可堆叠物品直接取回原实体——腐烂仍在继续。

**本次修改目标**：放置在地上的装饰品**永久保鲜**，取回时恢复正常的腐烂速率。

## 修改的 Mod 代码

### 改动 1：modmain.lua — `changeDecorationItem` 函数（放置时冻结腐烂）

```lua
-- 原代码（modmain.lua 第 28-29 行）：
	inst:AddTag("outofreach")--防偷
	inst:AddTag("decorationitem")--已放置的装饰品标签
end

-- 修改后：
	inst:AddTag("outofreach")--防偷
	inst:AddTag("decorationitem")--已放置的装饰品标签

	-- 通过腐烂速率×0实现放置保鲜（SetLocalMultiplier方案）
	-- 计时器仍在运行，但速率×0使得perishremainingtime不会减少
	-- _decoration_old_perish_mult 记录原始倍率，取回时恢复
	-- 用 nil 检查防止读档后 changeDecorationItem 重复调用时覆盖持久化的原始值
	if inst.components.perishable then
		if inst._decoration_old_perish_mult == nil then
			inst._decoration_old_perish_mult = inst.components.perishable:GetLocalMultiplier()
		end
		inst.components.perishable:SetLocalMultiplier(0)
	end
end
```

### 改动 2：modmain.lua — `OnSave`/`OnLoad` 持久化（存档后恢复保鲜状态）

```lua
-- 原代码（modmain.lua 第 56-71 行）：
			inst.OnSave = function(inst,data)
				if oldSaveFn~=nil then
					oldSaveFn(inst,data)
				end
				if inst:HasTag("decorationitem") then
					data.decorationitem=true
				end
			end
			inst.OnLoad = function(inst,data)
				if oldLoadFn~=nil then
					oldLoadFn(inst,data)
				end
				if data~=nil and data.decorationitem then
					changeDecorationItem(inst)
				end
			end

-- 修改后：
			inst.OnSave = function(inst,data)
				if oldSaveFn~=nil then
					oldSaveFn(inst,data)
				end
				if inst:HasTag("decorationitem") then
					data.decorationitem=true
					-- 持久化原始腐烂倍率，防止读档后丢失
					if inst._decoration_old_perish_mult ~= nil then
						data._decoration_old_perish_mult = inst._decoration_old_perish_mult
					end
				end
			end
			inst.OnLoad = function(inst,data)
				if oldLoadFn~=nil then
					oldLoadFn(inst,data)
				end
				if data~=nil and data.decorationitem then
					-- 先恢复持久化的原始腐烂倍率，再调 changeDecorationItem
					-- changeDecorationItem 检测到 _decoration_old_perish_mult 已有值就不覆盖
					if data._decoration_old_perish_mult ~= nil then
						inst._decoration_old_perish_mult = data._decoration_old_perish_mult
					end
					changeDecorationItem(inst)
				end
			end
```

**为什么 OnLoad 要先恢复 `_decoration_old_perish_mult` 再调 `changeDecorationItem`？**

读档流程中，`oldLoadFn` 先执行，恢复实体的所有原生状态。然后我们的代码执行。`changeDecorationItem` 内部的判断逻辑是：

```lua
if inst._decoration_old_perish_mult == nil then
    -- 首次放置：记录当前倍率
    inst._decoration_old_perish_mult = inst.components.perishable:GetLocalMultiplier()
end
```

如果不先恢复持久化的值，`_decoration_old_perish_mult` 为 nil，它会重新读取当前的 `GetLocalMultiplier()`，而当前可能已经被设为 0（如果 `oldLoadFn` 恢复了 `localPerishMultiplyer=1` 则不会出问题，但如果有其他 Mod 或环境因素修改过倍率则可能读到错误的值）。先恢复持久化值确保原始倍率不丢失。

### 改动 3：decoration_actions.lua — `RETRIEVEDECORATION` 非堆叠取回时恢复腐烂

```lua
-- 原代码（decoration_actions.lua 第 56-61 行）：
				elseif act.doer.components.inventory ~= nil and act.target.components.inventoryitem ~= nil then
					act.target.components.inventoryitem.canbepickedup=true
					act.target:RemoveTag("decorationitem")--已放置的装饰品标签
					act.doer.components.inventory:GiveItem(act.target,nil,pt)
					return true
				end

-- 修改后：
				elseif act.doer.components.inventory ~= nil and act.target.components.inventoryitem ~= nil then
					-- 取回时恢复原始腐烂速率倍率
					if act.target._decoration_old_perish_mult ~= nil
						and act.target.components.perishable then
						act.target.components.perishable:SetLocalMultiplier(
							act.target._decoration_old_perish_mult)
						act.target._decoration_old_perish_mult = nil
					end
					act.target.components.inventoryitem.canbepickedup=true
					act.target:RemoveTag("decorationitem")--已放置的装饰品标签
					act.doer.components.inventory:GiveItem(act.target,nil,pt)
					return true
				end
```

**注意**：堆叠物品分支（`act.target.once_stackable`）不需要恢复腐烂速率，因为那个分支走的是 `SpawnPrefab` 生成全新实体 + `Remove` 销毁旧实体。新实体自带默认 `localPerishMultiplyer=1.0`。

## 实现原理

### perishable 组件腐烂速率计算

`perishable` 组件的 `Update()` 函数（DST 本体 `components/perishable.lua`）中，腐烂计算公式为：

```lua
-- perishable.lua 第 127 行
modifier = modifier * self.localPerishMultiplyer

-- 第 132-133 行
self.perishremainingtime = math.min(self.perishtime,
    self.perishremainingtime - dt * modifier - additional_decay)
```

`localPerishMultiplyer` 初始值为 1.0（第 42 行）。通过 `SetLocalMultiplier(0)` 将其设为 0，则 `modifier = modifier × 0 = 0`，`perishremainingtime` 不再减少——等价于永久保鲜。

### 与 StopPerishing 方案的对比

| | `SetLocalMultiplier(0)` | `StopPerishing()` |
|---|---|---|
| 计时器状态 | 仍在运行（`updatetask` 不取消） | 完全暂停（`updatetask` 被 Cancel） |
| `perishremainingtime` | 保持不变（modifier×0） | 保持不变（不执行 Update） |
| 恢复方式 | `SetLocalMultiplier(原始值)` | `StartPerishing()` |
| 存档兼容 | 需要持久化原始倍率 | 不需要持久化额外值 |
| 鲁棒性 | 如果忘了恢复，保鲜效果永存 | 如果忘了恢复，保鲜效果永存 |

两种方案效果等价。此处选择 `SetLocalMultiplier` 是因为保持计时器结构不变，`perishremainingtime` 原样保留，恢复时无需额外同步。

## 中间遇到的错误及经验

### 错误 1：`GetPerishRateMultiplier` 方法不存在

**错误日志**：

```
[string "../mods/put/modmain.lua"]:37: attempt to call method
    'GetPerishRateMultiplier' (a nil value)
LUA ERROR stack traceback:
    ../mods/put/modmain.lua:37 in (upvalue) changeDecorationItem (Lua) <16-41>
       inst = 149638 - voltgoatjelly (valid:true)
```

**触发物品**：`voltgoatjelly`（伏特羊果冻），Warly 专属料理。该物品有 `perishable` 组件（食物），但没有 `preserver` 组件。

**错误原因**：`GetPerishRateMultiplier` 和 `SetPerishRateMultiplier` **不是 `perishable` 组件的方法**。以下是 DST 源码的验证过程：

```lua
-- perishable.lua：整个组件源码中只定义了以下公开方法：
-- SetPerishTime, SetLocalMultiplier, GetLocalMultiplier,
-- SetNewMaxPerishTime, StartPerishing, StopPerishing, IsPerishing,
-- Perish, GetPercent, SetPercent, ReducePercent, Dilute, AddTime,
-- IsFresh, IsStale, IsSpoiled, SetOnPerishFn, LongUpdate
--
-- 注意：没有 GetPerishRateMultiplier 或 SetPerishRateMultiplier！

-- 那这两个方法在哪？
-- 在 preserver 组件中（冰箱、盐盒等容器）
-- perishable:Update() 第 80 行：
modifier = owner.components.preserver:GetPerishRateMultiplier(inst) or modifier
--                                ^^^^^^^^^ 这是 preserver 组件的方法，
--                                           不是 perishable 的！
```

**教训：不要凭经验猜 API 名称。** `preserver` 和 `perishable` 是两个不同的组件，一个控制"容器如何影响内部物品的腐烂速率"，一个控制"物品自身的腐烂过程"。两个组件的 API 命名看似相近（`GetPerishRateMultiplier` vs `GetLocalMultiplier`），但归属完全不同。在 DST Mod 开发中，修改某个组件的行为时，**必须先读该组件的完整源码**确认方法名和参数签名，而不是凭其他组件的命名规律推断。

### 错误 2：方法名存在于思维中，但不存在于真实代码中

这个错误的根因是**把 `preserver:GetPerishRateMultiplier()` 的 API 名错记到了 `perishable` 组件上**。"腐烂速率"这个概念在 DST 中有两个层面：

```
容器层面 (preserver)                物品自身层面 (perishable)
─────────────────────────         ─────────────────────────
GetPerishRateMultiplier(inst)     GetLocalMultiplier()
SetPerishRateMultiplier(val)      SetLocalMultiplier(newMult)

作用：外部因素影响腐烂速率        作用：物品自身内部倍率
示例：冰箱×0.5、盐盒×0.25        初始值：1.0
                                  被谁调用：perishable:Update() 内部
                                  计算公式中：modifier *= localPerishMultiplyer
```

两个 API 都能实现"保鲜"的效果（乘以 0 就是冻结），但属于不同的组件。这次踩坑说明：**在 DST 的组件体系中，同义词的 API 名称可能属于不同组件，必须溯源确认。**
